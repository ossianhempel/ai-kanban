# Azure quickstart (~20 minutes)

Deploy AI Kanban on a **single Ubuntu VM** in your Azure subscription. This is the simplest supported path for teams (PGlite on local disk, HTTPS, ADO webhooks).

**Full reference:** [azure-vm.md](./azure-vm.md) (NSG details, backup, troubleshooting).

---

## What you will have

```text
kanban.contoso.com  →  Azure VM public IP
                          Docker: Caddy (:443) → ai-kanban (:3000)
                          Volume: /opt/ai-kanban/data  (database)
```

No App Service, no AKS, no fork required.

---

## 1. Azure Portal — create the VM

| Setting | Value |
|---------|--------|
| Image | **Ubuntu Server 22.04 LTS** |
| Size | **Standard_B2s** (2 vCPU, 4 GiB) |
| Auth | SSH public key |
| Inbound ports | **22**, **80**, **443** |

Note the **public IP**.

### DNS (required for HTTPS)

In your org DNS (or Azure DNS):

```text
kanban.contoso.com  A  <vm-public-ip>
```

Use your real hostname everywhere below instead of `kanban.contoso.com`.

### NSG (recommended)

- **22** — your IP only  
- **80, 443** — allow from internet (or corporate egress range)

---

## 2. VM — install Docker (once)

SSH in:

```bash
ssh azureuser@<vm-public-ip>
```

```bash
sudo apt-get update
sudo apt-get install -y curl ca-certificates
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker "$USER"
```

Log out and back in.

---

## 3. One-command install

```bash
curl -fsSL https://raw.githubusercontent.com/ossianhempel/ai-kanban/main/scripts/bootstrap-install.sh -o bootstrap-install.sh
chmod +x bootstrap-install.sh

sudo ./bootstrap-install.sh \
  --domain kanban.contoso.com \
  --allow-domain contoso.com
```

The script:

- Creates `/opt/ai-kanban` with `data/`, `.env`, `docker-compose.yml`, `Caddyfile`
- Generates `BETTER_AUTH_SECRET`, `AZURE_DEVOPS_WEBHOOK_SECRET`, `AIKANBAN_API_TOKEN`
- Pulls `ghcr.io/ossianhempel/ai-kanban:latest` (or set `--image` for your own registry)
- Starts **ai-kanban + Caddy** (Let’s Encrypt via Caddy)

Verify:

```bash
curl -s https://kanban.contoso.com/health
# {"ok":true}
```

Open `https://kanban.contoso.com` → sign up (first account is **admin**).

---

## 4. Work Azure extras

### Microsoft sign-in (recommended)

Follow [microsoft-sso.md](./microsoft-sso.md):

1. Entra app registration → redirect URI `https://kanban.contoso.com/api/auth/callback/microsoft`
2. Add client id/secret/tenant to `/opt/ai-kanban/.env`
3. Set `AUTH_PROVIDERS=microsoft` and `AUTH_EMAIL_PASSWORD_ENABLED=false`
4. `cd /opt/ai-kanban && docker compose up -d`

### Azure DevOps webhooks

ADO project → **Service hooks** → Web Hooks:

```text
https://kanban.contoso.com/api/webhooks/azure-devops?token=<AZURE_DEVOPS_WEBHOOK_SECRET>
```

Token is in `/opt/ai-kanban/.env` (generated by bootstrap).

### Agents (MCP)

Settings → **Agent integration** → copy Claude/Cursor config. Use `AIKANBAN_API_TOKEN` from `.env`.

Guide: [../agent-loop.md](../agent-loop.md)

## Updates

From the install directory (pull-based — no git):

```bash
cd /opt/ai-kanban
curl -fsSL https://raw.githubusercontent.com/ossianhempel/ai-kanban/main/scripts/update-stack.sh -o update-stack.sh
chmod +x update-stack.sh
./update-stack.sh
```

Or manually:

```bash
cd /opt/ai-kanban
docker compose pull && docker compose up -d
```

Pin a release tag instead of `latest`:

```bash
# In .env: AIKANBAN_IMAGE=ghcr.io/ossianhempel/ai-kanban:v0.1.0
docker compose pull && docker compose up -d
```

Publish tags from the upstream repo with a `v*` git tag (see `.github/workflows/docker-publish.yml`).

## HTTP-only / behind Azure load balancer

If TLS terminates at **Application Gateway** or similar:

```bash
sudo ./bootstrap-install.sh --http-only --install-dir /opt/ai-kanban
```

Edit `.env` so `WEB_ORIGIN` and `BETTER_AUTH_URL` match the URL users open, then `docker compose up -d`.

---

## Backup

```bash
cd /opt/ai-kanban
docker compose down
sudo tar -czf ~/ai-kanban-backup.tar.gz data .env
docker compose up -d
```

## When not to use a VM

| Option | Why skip for now |
|--------|------------------|
| Azure Container Apps | PGlite + network storage is unreliable |
| App Service | Weak persistent disk story for embedded DB |

Stick with **VM + Docker** until external Postgres is supported.

## Troubleshooting

| Issue | Fix |
|-------|-----|
| Certificate errors | DNS must point to VM before bootstrap; ports 80/443 open |
| GHCR pull 401 | Image private — `docker login ghcr.io` with a PAT |
| Sign-up blocked | Set `SIGNUP_ALLOWED_DOMAINS` or sign in with Microsoft SSO |
| Health unhealthy | `docker compose logs ai-kanban` — wait for migrations on first boot |
